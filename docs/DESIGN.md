# SchutzScript — 설계 문서 (v0.1)

> **한 줄 정의**
> AI 연동이 극도로 쉽고, **AI가 코드를 고치는 과정을 실시간으로 멋지게 보여주는** AI 네이티브 IDE.

작성일: 2026-07-09 · 상태: 초안(Draft) · 조직: [SchutzScript](https://github.com/SchutzScript)

## 1. 제품 정체성 (Product Identity)

Cursor / Windsurf 같은 "AI 네이티브 에디터" 카테고리에 속하지만, 우리의 **차별점은 단 하나에 집중**한다:

> **"AI의 편집 행위를 관찰 가능하고(observable), 아름답게(beautiful), 통제 가능하게(controllable) 만든다."**

대부분의 AI 코딩 툴은 결과(diff)만 던져준다. 우리는 **과정**을 보여준다. AI가 무슨 파일을 왜 여는지, 어떻게 고쳐 나가는지, 다음 계획이 무엇인지가 실시간 UI로 흐른다. 이게 제품의 심장이다.

### 설계 원칙
1. **Provider-agnostic** — 특정 AI 벤더에 묶이지 않는다. Claude / OpenAI / Gemini / 로컬 모델을 어댑터로 갈아 끼운다. (백엔드는 현재 미정이므로 추상화가 1순위)
2. **Observable by default** — AI의 모든 행동은 기본적으로 화면에 드러난다. "블랙박스 편집" 금지.
3. **Human-in-the-loop** — 모든 변경은 수락/거절/되돌리기 가능. AI는 제안하고 사람이 확정한다.
4. **Progressive fidelity** — 확장(Extension)에서 검증 가능한 것부터 시작하고, 확장 API로 불가능한 시각 효과가 명확해지면 포크(Fork)로 이전한다.

## 2. 핵심 UX — 실시간 편집 시각화 (네 기둥)

이 네 가지가 "멋진 실시간 표시"의 전부이며, 제품의 핵심 화면이다.

### 2.1 편집 과정 애니메이션 (Editing Animation)
AI가 코드를 "타이핑하듯" 흐르게 보여준다.
- **스트리밍 삽입**: 모델의 토큰 스트림을 그대로 에디터에 흘려서, 코드가 실제로 써지는 것처럼 렌더.
- **글로우/하이라이트**: 방금 바뀐 라인에 짧은 글로우 → 서서히 페이드. "여기가 방금 바뀌었다"를 눈이 자동으로 따라가게.
- **커서 고스트**: AI가 편집 중인 위치에 유령 커서 표시.
- ⚠️ **확장 한계**: VSCode 확장의 `TextEditorDecoration` API로 하이라이트/글로우는 가능. 하지만 "부드러운 타이핑 애니메이션"은 텍스트를 실제로 조금씩 삽입하거나 데코레이션 트릭으로 흉내내야 하며, 진짜 매끄러운 연출은 **포크 단계에서 렌더러 접근이 있어야 완성**된다.

### 2.2 변경 diff 시각화 (Diff Visualization)
무엇이 왜 바뀌는지 명확하게.
- 인라인 diff (추가=초록, 삭제=빨강, 수정=노랑) + **변경 사유 툴팁** ("이 함수는 null 체크가 없어 추가함").
- 라인 단위 **수락/거절 버튼** (CodeLens).
- 변경 그룹핑: 하나의 논리적 변경(예: "리팩터링")에 속한 여러 라인을 묶어서 한 번에 수락/거절.

### 2.3 AI 작업 상태·계획 패널 (Agent Activity Panel)
AI가 지금 뭘 왜 하는지 실시간 패널로.
- **현재 행동**: "`auth.ts`를 읽는 중…", "테스트 실행 중…", "3개 파일 수정 예정".
- **계획(Plan) 뷰**: 할 일 목록을 체크리스트로. 완료된 단계는 체크, 진행 중은 스피너, 예정은 회색.
- **툴 호출 로그**: 파일 읽기/쓰기/터미널 실행 등 AI의 모든 도구 사용을 타임라인으로.
- **토큰/비용 미터**: (BYOK 시) 이번 작업에 든 토큰·예상 비용.

### 2.4 멀티파일 동시 편집 뷰 (Multi-file Overview)
여러 파일을 한 번에 고칠 때 전체를 조망.
- **미니 파일 그리드**: 영향받는 파일들을 작은 카드로, 각 카드에 변경량(±N) 배지.
- **동시 진행 표시**: 어떤 파일이 편집 완료/진행 중/대기인지 색으로.
- 카드 클릭 → 해당 파일의 상세 diff로 점프.

### 3.2 AI Provider 추상화 (백엔드 미정 → 최우선 설계)

핵심 인터페이스. 어떤 벤더든 이 계약만 구현하면 붙는다.

```ts
interface AIProvider {
  readonly id: string;                // "claude" | "openai" | "gemini" | "local"
  readonly models: ModelInfo[];

  // 스트리밍이 실시간 UX의 전제 — 필수
  streamChat(req: ChatRequest): AsyncIterable<StreamEvent>;

  countTokens?(messages: Message[]): Promise<number>;
}

type StreamEvent =
  | { type: "text";      delta: string }               // 응답 텍스트 조각
  | { type: "tool_call"; name: string; args: unknown } // 도구 호출 요청
  | { type: "tool_result"; id: string; result: unknown }
  | { type: "edit";      file: string; patch: Patch }  // 코드 편집 (→ 애니메이션 트리거)
  | { type: "plan";      steps: PlanStep[] }           // 계획 갱신 (→ Agent 패널)
  | { type: "usage";     inputTokens: number; outputTokens: number }
  | { type: "done" };
```

> **권장 기본값**: 백엔드가 정해지면 **Claude API(BYOK)를 1순위 어댑터**로 구현 추천. 에이전트형 툴 사용(도구 호출·긴 컨텍스트·코드 편집)에서 현재 가장 강력하고, 스트리밍/툴 프로토콜이 이 아키텍처와 잘 맞음. 나머지 어댑터는 동일 인터페이스로 순차 추가.

### 3.3 편집 트랜잭션 모델
AI의 편집은 즉시 파일에 쓰이지 않고 **트랜잭션**으로 관리된다.
- `pending` → 화면엔 보이지만 디스크 미반영, 수락/거절 대기.
- `accepted` → 실제 파일 반영.
- `rejected` → 폐기.
- **되돌리기(Undo)**: 트랜잭션 단위로 전체 롤백. AI 작업 하나를 통째로 취소 가능.

이 모델이 있어야 2.2의 수락/거절 UI와 "안전한 관찰 가능성"이 성립한다.

- Provider 추상화 + 첫 어댑터 1개

- Agent 활동 패널 Webview (2.3)
- 멀티파일 오버뷰 Webview (2.4)
- **검증 질문**: 확장 API로 어디까지 "멋지게" 되는가? 무엇이 답답한가?

### Phase 2 — 포크(Fork)로 이전
> 목표: 확장으로 불가능했던 시각 효과를 위해 Code-OSS 포크. 진짜 차별화 완성.
- Code-OSS / VSCodium 기반 브랜딩·빌드 파이프라인
- 에디터 렌더러 레벨의 부드러운 타이핑 애니메이션 (2.1 완성)
- 확장 코드를 코어 기능으로 통합
- 배포(자동 업데이트, 서명 등)

### Phase 3 — 확장·생태계
- 멀티 프로바이더 완성
- 컨텍스트 관리(코드베이스 인덱싱/RAG)
- 협업, 세션 공유 등

## 6. 리스크 & 열린 질문

**리스크**
- **R1. 확장 API의 시각 표현 한계** — 2.1의 매끄러운 애니메이션이 확장에서 기대만큼 안 나올 수 있음. → Phase 1에서 조기 검증, 안 되면 Phase 2 앞당김.
- **R2. 포크 유지보수 부담** — VSCode 업스트림 추종 비용. VSCodium 사례 참고로 최소화.
- **R3. 실시간 스트리밍 편집의 충돌** — 사용자가 편집 중인데 AI도 편집하면? → 트랜잭션 모델 + 락/머지 정책 필요.

**결정해야 할 열린 질문**
- Q1. 첫 타깃 AI 백엔드는? (권장: Claude BYOK)
- Q2. "SchutzScript"라는 이름의 의미 — 자체 스크립트 언어 요소가 있나, 순수 IDE 브랜드인가?
- Q3. 오픈소스 라이선스 정책 (Code-OSS는 MIT, 포크 시 상표 이슈 주의)
- Q4. 배포 타깃 OS 우선순위 (Windows 우선? 크로스플랫폼?)
